# Лабораторные работы — OOP

## lab1 — C++ (Консоль)

### Требования
- Выбрать и согласовать с преподавателем предметную область.
- Создать иерархию классов: ≥1 родительский + ≥2 дочерних.
- Каждый класс: ≥2 член данных, ≥2 собственных методов; дочерние: ≥2 унаследованных + ≥2 перекрытых.
- Продемонстрировать инкапсуляцию, наследование, полиморфизм.
- Работа с объектами — только через переменные родительского типа.
- Каждый класс в отдельном файле.
- Отчёт: описание предметной области, схема иерархии, описание методов, выводы.

### Проверить
1. Базовый класс абстрактный (`virtual ... = 0`)
2. Наследники имеют ≥3 конструкторов
3. Доступ к полям через геттеры/сеттеры
4. Массивы/векторы родительского типа + вызов всех методов
5. Отчёт с диаграммой классов

### Преимущества реализации
- Ручное управление памятью через `shared_ptr` — явный контроль времени жизни объектов
- Виртуальные таблицы (vtable) на этапе компиляции — минимальный оверхед полиморфизма
- Отсутствие рантайма — максимальная производительность
- CMake — кроссплатформенная сборка без привязки к IDE

## lab2 — C# Desktop Application

### Требования
- Графический интерфейс (десктопное приложение).
- Хранение данных в БД (PostgreSQL).
- Все требования lab1 в силе.
- CRUD + все методы из lab1.

### Проверить
1. Базовый класс абстрактный
2. Каждый класс отдельно
3. Геттеры/сеттеры
4. CRUD, данные не скопом в одной таблице
5. Отчёт: диаграмма классов (включая формы), особенности подключения к БД

### Преимущества реализации (Desktop)
- Полный контроль над интерфейсом — нет ограничений браузера (CORS, JS-песочница)
- Нативное взаимодействие с ОС — drag & drop из проводника, диалоги выбора файлов
- Офлайн-режим — приложение работает без сети, БД на локальном сервере
- Богатая типовая графика — кастомная отрисовка, градиенты, анимация без CSS/Canvas
- Реактивность — мгновенный отклик на действия пользователя без сетевых задержек
- Единый язык (C#) — модели, логика и UI в одном проекте, общие типы данных

## lab3 — Java Spring Boot

### Требования
- Язык Java
- PostgreSQL + Hibernate (JPA)
- Maven
- ≥3 связанных таблицы в 3НФ
- Связь не по наследованию (ManyToMany через playlist_media)
- Аннотации JPA + Lombok
- Web-интерфейс (Thymeleaf)
- `CrudRepository` для CRUD

### Дополнительно
- Фильтрация, поиск, статистика
- Слияние плейлистов (merge) и клонирование (clone) через `@Transactional`
- REST API (`/api/stats`, `/api/health`)

### Преимущества реализации
- JPA + `@Inheritance(JOINED)` — объектно-реляционное отображение иерархии без SQL
- `@Transactional` — атомарные операции на уровне сервиса (слияние плейлистов)
- Thymeleaf — серверный рендеринг HTML без разделения на фронт и бэк
- Lombok `@Data` — генерация геттеров/сеттеров/toString одной аннотацией
- Spring Boot autoconfiguration — минимум boilerplate для запуска веб-приложения
- Единый контейнер (монолит) — простота деплоя и отладки

## lab4 — Go + Python (Распределённая система)

### Требования
- SQLAlchemy (Python) — слой доступа к данным
- Go — бизнес-логика + веб-сервер
- PostgreSQL (та же схема, что в lab3)
- UI — HTML (можно из lab3)

### Технические требования
- Модели Python строго соответствуют таблицам БД
- Repository pattern на Python
- Go: фильтрация, сумма/среднее/количество, валидация, TTL-кэш (5 мин), поиск по разным полям
- UI: просмотр, редактирование, удаление, добавление, все методы lab1, валидация на клиенте, выпадающие списки для FK, связанные сущности

### Преимущества реализации
- Разделение ответственности: Python отвечает только за ORM и CRUD, Go — за бизнес-логику и кэширование
- TTL-кэш (5 мин) на Go — снижение нагрузки на Python/БД для часто запрашиваемых данных
- Fan-out goroutines + channel — конкурентная загрузка плейлистов при слиянии
- Валидация бизнес-правил на Go до отправки в Python — защита от невалидных данных
- Асинхронный health-check (JS → Go → Python) — мониторинг распределённой системы
- Разные языки для разных задач: Python (быстрое прототипирование ORM), Go (производительность и конкурентность)
- Легковесные контейнеры: каждый сервис можно масштабировать независимо
