# Полный конспект лекции: Объектно-ориентированное программирование (ООП)

## Введение и мотивация
ООП — парадигма программирования, ориентированная на моделирование предметной области посредством объектов, которые инкапсулируют состояние и поведение. ООП помогает проектировать масштабируемые, поддерживаемые и расширяемые системы.

Цели лекции:
- понять ключевые концепции ООП;
- сравнить реализацию в C++, Java и Python;
- изучить практики проектирования (SOLID, композиция vs наследование);
- проработать примеры кода и подводные камни.

---

## Парадигмы программирования (обзор)
- Процедурное (структурное) — функции и процедуры.
- Функциональное — ф-я как гражданин первого класса, неизменяемость.
- Логическое — правила и выводы (Prolog).
- Автоматное — конечные автоматы и их применение.
- Объектно-ориентированное — объекты и классы.
- Событийно-ориентированное — обработка событий.
- Агентно-ориентированное — агенты с сенсорами и эффекторами.

---

## Базовые понятия ООП
- Класс — шаблон/тип, описывающий структуру данных и поведение.
- Объект — экземпляр класса с конкретным состоянием.
- Поле (атрибут) — хранит состояние объекта.
- Метод — операция, которая может выполнять объект.
- Интерфейс/контракт — набор публичных методов, которые класс обязуется реализовать.
- Инкапсуляция, абстракция, наследование, полиморфизм — четыре краеугольных принципа.

---

## Элементы объектной модели и принципы
- Абстракция: выделение существенных свойств и сокрытие деталей реализации.
- Инкапсуляция: контроль доступа к состоянию через интерфейсы (getter/setter, методы).
- Наследование: повторное использование кода, иерархии типов.
- Полиморфизм: единый интерфейс для разных реализаций (динамический dispatch).
- Модульность: разделение на компоненты с минимальной связностью.
- Контроль типов: компилятор/интерпретатор помогает обнаруживать ошибки.
- Персистентность: сохранение состояния объектов (сериализация, БД).
- Параллелизм/Concurrency: безопасный доступ к объектам в многопоточном окружении.

---

## Почему ООП: преимущества и недостатки
Преимущества:
- Переиспользование кода (наследование, композиция).
- Улучшенная структура и читаемость.
- Упрощённое моделирование сложных доменных областей.
- Поддержка инкапсуляции и абстракции.

Недостатки:
- Порог вхождения выше, чем у процедурного подхода.
- Избыточность для простых задач.
- Возможно снижение производительности (высокая абстракция).
- Риск избыточной сложности при неправильном проектировании (глубокие иерархии).

## SOLID и хорошие практики проектирования (кратко)
- S — Single Responsibility Principle: каждый класс имеет одну ответственность.
- O — Open/Closed: открыт для расширения, закрыт для модификации.
- L — Liskov Substitution: подклассы должны заменять базовые классы без нарушения корректности.
- I — Interface Segregation: узкие интерфейсы лучше, чем монолитные.
- D — Dependency Inversion: зависимости направлены на абстракции, не на конкретные реализации.

Дополнительно:
- Предпочитать композицию наследованию, когда возможно.
- Минимизировать видимость (приватные поля, публичные интерфейсы).
- Явные интерфейсы и слабая связность (loose coupling).
- Тестируемость: писать тесты для поведения, а не для реализации.

---

## Области видимости / Access Modifiers (сравнение)
- public — доступ отовсюду; публичный интерфейс.
- protected — доступ внутри класса и его потомков (и/или внутри пакета, зависит от языка).
- private — доступ только внутри класса; максимальная инкапсуляция.
- package / internal / module-level — средний уровень доступа в некоторых языках.

Замечание: семантика доступа и соглашения (например, '_' и '__' в Python) различаются по языкам.

---

## Наследование vs Композиция
- Наследование: «is-a» — используется, когда существует явная иерархия типов.
- Композиция: «has-a» — предпочтительна для повторного использования и снижения связности.
- Избегать глубокой иерархии; предпочитать интерфейсы и делегирование.

---

## Полиморфизм и динамическое связывание
- Статическое (compile-time) полиморфизм: перегрузка функций, шаблоны/generics.
- Динамическое (run-time): виртуальные методы, интерфейсы.
- Принцип: код работает с абстракциями, конкретные реализации подставляются во время выполнения.

---

## Управление памятью и жизненный цикл объектов
- Конструкторы/деструкторы (C++).
- Garbage collection (Java, Python) — автоматическое освобождение памяти.
- RAII (C++) — связывание ресурса с временем жизни объекта.
- Copy / Move семантика (C++) — важно для управления ресурсами.
- Smart pointers (C++): unique_ptr, shared_ptr, weak_ptr.

## Языковые особенности и примеры

### C++: ключевые моменты
- this — указатель на текущий объект внутри нестатических методов: this->member.
- Smart pointers:
  - unique_ptr<T> — эксклюзивное владение, неповторяемое копирование.
  - shared_ptr<T> — подсчет ссылок, совместное владение.
  - weak_ptr<T> — неблокирующая ссылка для избежания циклических зависимостей.
- RAII: ресурсы (файлы, память) закрываются в деструкторах.
- Конструкторы копирования и перемещения: определить или запретить при наличии ресурсов.
- Виртуальные функции и виртуальный деструктор для корректного удаления через базовый указатель.
- Diamond problem при множественном наследовании — решение через виртуальное наследование.
- Примеры ошибок: утечки памяти при raw pointers, двойное удаление, неопределенное поведение при использовании освобожденных объектов.

Короткий пример (идея):
- уникальный владелец (unique_ptr), shared_ptr с подсчетом ссылок, использование this в методе printInfo.

### Java: ключевые моменты
- this — ссылка на текущий объект (не указатель).
- Все объекты — ссылочные типы; примитивы хранятся отдельно.
- Garbage Collector освобождает память автоматически.
- final — предотвращает переопределение/наследование (класс/метод/переменная).
- equals() и hashCode() — важны при работе с коллекциями.
- Сериализация (Serializable) и transient.
- Исключения — checked vs unchecked.
- Reflection и ClassLoader — динамика создания и загрузки типов.

Особенности:
- Нет примитивного управления памятью (new/delete отсутствуют).
- Потенциальные паузы из-за GC, хотя современные JVM минимизируют влияние.
- Ссылочная семантика: присвоение копирует ссылку, не объект.

### Python: ключевые моменты
- Динамическая типизация и duck typing: «если он ходит как утка…».
- Соглашения по доступу: _protected (convention), __private (name mangling).
- Все объекты на heap, ссылки передаются по ссылке.
- Миксин-классы и множественное наследование.
- Декораторы (@property, @staticmethod, @classmethod).
- Контекстные менеджеры (with) и протоколы __enter__/__exit__ для RAII-подобных сценариев.

## Особенности и подводные камни (советы)
- Не злоупотребляйте наследованием: используйте composition over inheritance.
- Избегайте глубоких иерархий и «божественных» классов (God objects).
- Делайте классы небольшими и специализированными.
- Инкапсулируйте состояние; не раскрывайте реализацию через публичные поля.
- Проектируйте API стабильно: изменения в публичном интерфейсе — дорогостоящи.
- Документируйте предположения и invariants класса (предварительные условия, постусловия).
- В многопоточном коде — защищайте состояние через синхронизацию/immutable объекты.

---

## Конкурентность и объекты
- Immutable объекты безопасны для чтения из нескольких потоков.
- Для изменяемых объектов — использовать mutex/lock, атомарные операции.
- В Java: synchronized, java.util.concurrent (atomic, locks, concurrent collections).
- В C++: std::mutex, std::atomic, lock_guard, уникальные/разделяемые блокировки.
- Проектирование API: минимизировать время удержания блокировок, избегать deadlock.

- Сериализация объектов в строки/JSON/бинарный формат.

- Важность версионирования при сериализации/обновлениях формата.

## Тестирование и отладка
- Юнит-тестирование классов (входы/выходы, граничные условия).

- Инварианты класса и предусловия — проверять в тестах.
- Инструменты профилирования и анализатор утечек памяти (Valgrind для C++, инструменты JVM).
